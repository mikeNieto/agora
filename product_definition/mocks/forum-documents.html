Documento de Arquitectura de Alto Nivel (HLD)

Plataforma E-commerce B2B — Migración a Microservicios

1. Resumen Ejecutivo

Este documento describe la arquitectura de microservicios propuesta para migrar la plataforma e-commerce B2B monolítica (PHP/Laravel) a un sistema distribuido de alta disponibilidad. La migración se realizará en tres fases durante un período de 18 meses, utilizando el patrón Strangler Fig para garantizar continuidad del negocio.

2. Objetivos y Restricciones

2.1 Objetivos

  • SLA del 99.9% (máximo 8.7 horas de downtime anual)
  • Soportar 50,000 pedidos/hora en picos de campaña
  • Gestionar 500,000+ SKUs con precios por cliente
  • Reducir el tiempo de despliegue de 2 horas a menos de 15 minutos

2.2 Restricciones

  • Presupuesto: $2,000–$10,000/mes en AWS
  • Equipo: Sin especialistas K8s — uso de servicios gestionados
  • Stack existente: PHP/Laravel en backend, React en frontend
  • Plazo: 18 meses para migración completa

3. Arquitectura Propuesta

💬 2 comentarios

3.1 Capa de Infraestructura

La arquitectura base utiliza AWS ECS Fargate como plataforma de contenedores, eliminando la gestión de servidores y simplificando las operaciones para el equipo actual. Un AWS API Gateway centraliza el enrutamiento y actúa como proxy hacia el monolito durante las fases de transición.

3.2 Fases de Migración

Fase 1 — Infraestructura Base (Meses 1–4)

  • Desplegar API Gateway + ECS Fargate
  • Extraer Catalog Service (500k SKUs, ElastiCache para precios)
  • Extraer Pricing Service (lógica de precios por cliente)
  • Monolito continúa manejando pedidos, usuarios y sesiones

Fase 2 — Core Business (Meses 5–8)

  • Extraer Order Processing Service con cola SQS
  • Extraer Inventory Service
  • Extraer Customer Service con JWT básico
  • Implementar Event Sourcing para pedidos

Fase 3 — Consolidación (Meses 9–18)

  • Migrar IAM completo con OAuth 2.0
  • Descomisionar el monolito progresivamente
  • Implementar observabilidad completa (Datadog/CloudWatch)
  • Evaluar migración a EKS para equipos que hayan madurado

4. Decisiones Técnicas Clave

💬 1 comentario

Las decisiones fundamentales adoptadas por consenso entre los agentes:

  • ECS Fargate sobre EKS: Menor overhead operacional para el equipo actual. K8s se incluye como hoja de ruta a largo plazo.
  • Strangler Fig: Migración gradual sin big bang, permitiendo validar cada servicio antes de continuar.
  • JWT en Fase 2 (no Fase 1): Evita refactoring del frontend React en las primeras etapas críticas. Las sesiones PHP se mantienen durante Fase 1.
  • SQS para Order Processing: Garantiza durabilidad de pedidos y permite escalar workers independientemente.
  • ElastiCache para Catalog: Reducción de latencia en consultas de precios de ~200ms a <5ms.
